## Summary

Today's session delivered two major features that address critical retention and onboarding gaps: **push notifications for overdue tasks** and a **local-to-cloud migration wizard**. Both features were built, tested via `npm run build`, and pushed to `main`.

The major themes were:

## Features implemented

### 1. Push notifications for overdue tasks (Feature #5)

**Problem:** Without notifications, the user has to remember to open the app — which is exactly the problem the app is supposed to solve.

**Solution:** Custom service worker with periodic background sync that checks for overdue maintenance tasks and shows a notification even when the app is closed.

Changes:
- Switched `vite-plugin-pwa` from `generateSW` to `injectManifest` mode for custom service worker logic
- Created `src/sw.ts` — handles `periodicsync` event (tag: `check-overdue-tasks`), reads task data from Cache API, shows notification if overdue tasks exist, handles notification click to open `/maintenance`
- Created `src/composables/useNotifications.ts` — manages notification permission state, `requestPermission()` triggers periodic sync registration (24h interval)
- Added Cache API mirror in `src/composables/useMaintenanceData.ts` — writes a task snapshot to `overdue-check-data` cache on every change so the service worker can access it (service workers can't read localStorage)
- Added "Benachrichtigungen" section in Settings with three states: activation button, green confirmation, amber "blocked" warning

**Browser support:**
- Periodic Background Sync: Chrome/Edge Android (installed PWAs)
- iOS/Safari/Firefox: permission prompt works, but background checks won't fire (no periodic sync support)
- Server-side push via Supabase Edge Functions can be added later

Related files:

### 2. Local-to-cloud migration wizard (Feature #6)

**Problem:** A user who starts locally and later signs up loses their data. Vehicles, tasks, logs, places, and playlists stay orphaned in localStorage while the app starts reading from empty Supabase tables.

**Solution:** After first sign-in, a global dialog detects local data and offers one-click migration to the cloud account.

Changes:
- Created `src/services/storage/migrationService.ts` — core migration engine:
  - `hasLocalData()` — checks if localStorage has any vehicles, tasks, places, or playlists
  - `wasMigrated()` — checks flag to prevent double-migration
  - `getLocalDataPreview()` — reads all local data for the preview dialog
  - `migrateLocalToCloud()` — creates vehicles via RPC (builds ID mapping), remaps `vehicleId` in tasks/logs, uploads places/playlists as user-level data, clears localStorage
  - `restoreUserDataFromCloud()` — on new device sign-in, fetches places/playlists from cloud and writes to localStorage
- Created `src/components/MigrationWizard.vue` — 4-step modal dialog: preview → migrating spinner → success → error with retry
- Created `src/services/storage/supabaseUserDataRepository.ts` — simple key-value Supabase CRUD for user-level JSON data (`get`, `set`)
- Created `supabase/patch-user-data-table.sql` — `user_data` table with RLS (applied to Supabase)
- Migration trigger lives in `src/components/AppShell.vue` (global level) — fires immediately after sign-in on any page

**Data migrated (5 types):**
| Data | Target | ID handling |
|------|--------|-------------|
| Vehicles | `vehicles` table via RPC | New UUIDs, mapping built |
| Tasks | `maintenance_tasks` table | Vehicle IDs remapped, task IDs preserved |
| Logs | `maintenance_logs` table | Vehicle IDs remapped, log IDs preserved |
| Places | `user_data` table (key: `saved-places`) | IDs preserved as JSON |
| Playlists | `user_data` table (key: `playlist-shortcuts`) | IDs preserved as JSON |
